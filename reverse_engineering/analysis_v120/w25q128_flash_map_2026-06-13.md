# W25Q128 SPI-flash map — live dump from Unit 2 (2026-06-13)

**Method:** dumped directly from the device over the USB debug shell (`flash dump`,
clean binary via `scripts/flash_capture.py`) and parsed the FAT12 structures offline.
No reflash; the running firmware already exposes `flash jedec` / `flash dump`.

This was a resolve-loop detour off the `master_init` decode: the `flash_cfg_cal` region
(R0 floor) showed stock loads saved settings + a calibration table from SPI flash at boot
(`spi_flash_read_config`). Question: **is the factory calibration recoverable from the
W25Q?** Answer below.

## Chip + layout

- **JEDEC `EF 40 18`** = Winbond W25Q128 (16 MB), matches `CLAUDE.md`. Alive, reads clean.
- **Two FAT12 volumes, no MBR, exactly filling 16 MB** (4096 B/sector = the W25Q erase unit):

| Flash range | Size | FatFS vol | BPB | Contents |
|---|---|---|---|---|
| `0x000000`–`0x1FFFFF` | 2 MB | **`3:` (base 0, `uVar2==3` in `FUN_0802d014`)** | rsvd 1, 2×FAT 1 sec, 512 root ent | **`System file/`** — UI assets |
| `0x200000`–`0xFFFFFF` | 14 MB | **`2:` (base `0x200000`, `uVar2==2`)** | rsvd 1, 2×FAT 2 sec | Screenshot/other store |

  (The FatFS volume numbering is inverted from intuition — confirmed against the
  `FUN_0802d014` cache function: vol 3 → flash base 0, vol 2 → flash base `0x200000`.)

### Volume `3:` (System file) geometry
- FAT @ `0x1000`, root dir @ `0x3000`, data region @ `0x7000`, cluster N → `0x7000+(N-2)·0x1000`.
- Root: `System Volume Information/` + **`System file/`** (dir, cluster 5, chain `5 → 355`).
- `System file/` holds **163 entries: 160 × `*.JPG` + 1 × `9999.BIN` + `.`/`..`**.
  - The JPEGs (`1.JPG`…`154.JPG`, etc.) are the UI / boot-logo image assets — directly
    confirms the `scope_render_monsters` finding that `FUN_08030524`/`FUN_08031f20` are a
    JFIF/Huffman decoder pulling UI graphics from SPI flash (not waveform render).

## Headline finding: **the factory calibration is NOT in the W25Q (on this unit)**

- The only non-image file, **`9999.BIN`** (the config filename — stock string
  `3:/System file/9999.bin` @ `0x080b5841`), is **cluster 0, size 0 = an empty factory
  placeholder** (it has a real creation timestamp but was never written with data).
- This **confirms by direct flash read** what `external_flash_runtime_path_map_2026_04_08.md`
  inferred from RE: `9999.BIN` is empty, has no direct xrefs, and is not the cal store.
- The two volumes exactly cover all 16 MB → **no raw (non-FAT) region** could hold a
  hidden cal blob either. The W25Q content is: UI JPEGs + screenshots + one empty config
  placeholder. **No calibration data anywhere on the chip.**

## Where the cal therefore lives (narrowed by elimination)

The scope gain/offset cal table (120 B → RAM `0x20000358`) and saved device config are
**not on the W25Q**. Remaining sources:
1. **MCU internal flash `0x08006000`** (phase4 `saved_config`) — **overwritten by our own
   firmware** (our app links at `0x08004000+`), so stock's copy is gone on Unit 2. Would
   need a pristine-stock unit to recover.
2. **The FPGA bitstream** — `CLAUDE.md`'s revised (2026-06-10) leading hypothesis for the
   *meter* autorange/cal. Consistent with the W25Q being cal-free.

**Implication for the `flash_cfg_cal` R0 floor:** faithfully reimplementing the stock
boot-cal restore is blocked not by missing code understanding but by the **cal source being
absent on our bench unit**. To populate it we'd need either a stock-flash dump from a
pristine unit (the MCU `0x08006000` sector) or the cal extracted from the FPGA bitstream.

## Reusable tooling produced
- `scripts/flash_capture.py` — clean binary capture from the `flash dump` shell command
  (handles the `FLASHDUMP <len>` framing; the obvious off-by-one is consuming the header
  `\r\n` before the raw bytes).
- The FAT12 BPB/dir/FAT-chain parsing snippets in this session reconstruct the filesystem
  offline from targeted dumps — no full 16 MB dump needed.
